记得。我们当时做的是：**把原生网络出口包一层 `safe_network`，每次网络请求必须带 `NetworkPurpose`，SafeMode 下按 purpose 放行或阻断。**

核心文件是：

```text
utils/safety/src/safe_network.rs
```

目前主要有这几类函数。

---

## 1. 最核心的把关函数

### `safe_network::ensure_allowed(...)`

用途：  
**只检查这个网络目的是否允许，不负责真正发送请求。**

典型用法：

```rust
safe_network::ensure_allowed(NetworkPurpose::Other)?;
```

它适合放在这些地方前面：

```text
TcpListener::bind
TcpStream::connect
connect_async
connect_async_tls_with_config
axum::serve
本地/远程 websocket
proxy listen / connect
```

也就是说，凡是不能直接包成 reqwest request 的网络入口/出口，就先手动过这一关。

你现在的扫描结果里已经有很多这种用法，比如 websocket transport、exec-server、network-proxy 等地方都在调用 `safe_network::ensure_allowed(NetworkPurpose::Other)`。

---

## 2. 异步 reqwest 请求替代函数

### `safe_network::send(...)`

用途：  
替代原生：

```rust
request.send().await
```

新写法：

```rust
safe_network::send(NetworkPurpose::ModelApi, request).await
safe_network::send(NetworkPurpose::ChatGPTAuth, request).await
safe_network::send(NetworkPurpose::Other, request).await
```

对应关系：

```text
原生：
reqwest::RequestBuilder::send().await

替代：
safe_network::send(NetworkPurpose::X, request_builder).await
```

你的扫描结果里能看到很多位置已经改成了：

```rust
safe_network::send(NetworkPurpose::Other, request)
safe_network::send(NetworkPurpose::ModelApi, request)
safe_network::send(NetworkPurpose::ChatGPTAuth, request)
```

比如 `codex-api` 文件上传使用 `NetworkPurpose::ModelApi`，登录相关使用 `NetworkPurpose::ChatGPTAuth`，plugin / skill / remote / update 这类多数先归到 `NetworkPurpose::Other`。

---

## 3. 阻塞式 reqwest 请求替代函数

### `safe_network::blocking_send(...)`

用途：  
替代原生阻塞请求：

```rust
request.send()
```

新写法：

```rust
safe_network::blocking_send(NetworkPurpose::Other, request)
```

对应关系：

```text
原生：
reqwest::blocking::RequestBuilder::send()

替代：
safe_network::blocking_send(NetworkPurpose::X, request_builder)
```

你之前的扫描里有 `tui/tooltips.rs`、`responses-api-proxy`、`tui/pets/asset_pack.rs` 等地方出现过 `blocking_send`。

---

## 4. Codex 客户端封装函数

### `send_with_purpose(...)`

这个不是最底层门禁，而是对 Codex 自己的 request builder 做了一层更顺手的封装。

典型形式：

```rust
.send_with_purpose(NetworkPurpose::ModelApi)
.send_with_purpose(NetworkPurpose::ChatGPTAuth)
```

对应关系：

```text
原生：
builder.send().await

Codex 封装：
builder.send_with_purpose(NetworkPurpose::ModelApi).await
```

从扫描结果看，`codex-client/src/default_client.rs` 里 `send_with_purpose` 内部最终会调用：

```rust
safe_network::send(purpose, builder).await
```

所以它本质上还是走 `safe_network::send`。

---

## 5. NetworkPurpose 枚举

目前核心分级是：

```rust
NetworkPurpose::ChatGPTAuth
NetworkPurpose::ModelApi
NetworkPurpose::Other
```

含义：

```text
ChatGPTAuth
= 登录、刷新 token、账号认证相关。

ModelApi
= 模型推理、Responses API、文件上传到模型接口等必要请求。

Other
= 其他所有网络行为。
```

SafeMode 下我们之前的原则是：

```text
允许：
ChatGPTAuth
ModelApi

默认阻断：
Other
```

你之前的 safe_network 片段里也显示了这个判断逻辑：`allowed_in_safe_mode()` 只匹配 `ChatGPTAuth | ModelApi`，如果 SafeMode 开启且 purpose 不允许，就拒绝。

未来如果做 Tavily，要加：

```rust
NetworkPurpose::TavilySearch
```

但它不能简单等同 `Other`，必须受 Tavily 配置、日志、风险等级和确认规则控制。

---

## 6. 具体替代清单

### A. HTTP 请求

```text
原生：
request.send().await

替代：
safe_network::send(NetworkPurpose::X, request).await
```

---

### B. Blocking HTTP 请求

```text
原生：
request.send()

替代：
safe_network::blocking_send(NetworkPurpose::X, request)
```

---

### C. CodexRequestBuilder 请求

```text
原生：
builder.send().await

替代：
builder.send_with_purpose(NetworkPurpose::X).await
```

内部仍应走：

```rust
safe_network::send(...)
```

---

### D. WebSocket 连接

```text
原生：
connect_async(...)
connect_async_with_config(...)
connect_async_tls_with_config(...)

替代：
safe_network::ensure_allowed(NetworkPurpose::X)?;
connect_async(...)
```

也就是 WebSocket 不一定直接包成 `safe_network::send`，但连接前必须先 `ensure_allowed`。

---

### E. TCP 连接

```text
原生：
TcpStream::connect(...)
tokio::net::TcpStream::connect(...)

替代：
safe_network::ensure_allowed(NetworkPurpose::X)?;
TcpStream::connect(...)
```

---

### F. TCP / HTTP 监听

```text
原生：
TcpListener::bind(...)
StdTcpListener::bind(...)
axum::serve(...)
Server::http(...)

替代：
safe_network::ensure_allowed(NetworkPurpose::X)?;
TcpListener::bind(...)
```

这类属于“网络入口”，不是请求出口，也必须过门禁。

---

### G. Network Proxy

```text
原生：
proxy listen / connect / forward

替代：
safe_network::ensure_allowed(NetworkPurpose::Other)?;
```

SafeMode 下，`NetworkPurpose::Other` 默认应该被挡掉，所以 proxy 也就不会偷偷打开。

---

## 7. 你可以记成这 3 个主函数

最重要就是这三个：

```rust
safe_network::ensure_allowed(NetworkPurpose::X)
safe_network::send(NetworkPurpose::X, request).await
safe_network::blocking_send(NetworkPurpose::X, request)
```

然后 Codex 自己还有一个便利封装：

```rust
send_with_purpose(NetworkPurpose::X)
```

最终原则：

```text
能包 request 的，用 safe_network::send / blocking_send。
不能包 request 的，比如 websocket、listener、proxy、tcp connect，就先 safe_network::ensure_allowed。
Codex 模型 API 客户端，用 send_with_purpose，但内部必须落到 safe_network::send。
```
