
## 1. Codex 是纯 Rust 写的吗？

**你现在打包的 CLI 主程序，基本可以按 Rust 项目来处理。**

也就是这个：

```text
codex-rs\target\release\codex.exe
```

是 Rust 编译出来的。

但不能说整个仓库 100% 只有 Rust。仓库里可能还有：

```text
Markdown
TOML / YAML / JSON
脚本
测试资源
内置文档
历史残留文件
```

所以安全判断要分两层：

```text
1. release exe 的主执行链路：主要审 Rust。
2. 仓库里可能被调用的脚本 / 配置 / 资源：也要防止被自动加载或执行。
```

这也是为什么我们之前封 `skills / plugins / connectors / .codex`，不是因为它们一定会执行，而是因为它们可能成为非 Rust 主链路的扩展入口。

## 2. 能不能像网络一样，给所有本地文件读写也发“门禁卡”？

**能，而且应该做。**

网络的方案是：

```text
所有网络请求
=> safe_network
=> NetworkPurpose
=> SafeMode 审核
```

本地文件也可以做成：

```text
所有本地文件访问
=> safe_fs
=> FsPurpose
=> CToolScope / SafeMode 审核
```

比如设计：

```rust
enum FsPurpose {
    ReadCoolSystemConfig,
    ReadCoolSessionConfig,
    ReadToolInputFile,
    SearchToolInputPath,
    WriteToolOutputFile,
    WriteCommandRequestLog,
    WriteSessionRollout,
    ReadSessionRollout,
    AuthStateReadWrite,
}
```

然后所有读写都必须走：

```rust
safe_fs::read_to_string(...)
safe_fs::write(...)
safe_fs::create_dir_all(...)
safe_fs::read_dir(...)
safe_fs::open(...)
```

现在代码里还存在裸 `std::fs` 调用。比如配置读取现在直接 `std::fs::read_to_string(...)`，命令日志也直接 `create_dir_all`、`write`、`OpenOptions`。

所以答案是：

```text
可以做。
但当前还没完全做完。
```

## 3. 纯 Rust 开源工具，联网能不能做到“全局扫描，一个不漏”？

**接近可以，但不能只靠搜索字符串保证 100%。**

如果目标是 Rust 主程序，可以全局扫这些：

```text
reqwest
hyper
ureq
surf
isahc
tokio::net
std::net
TcpStream
UdpSocket
UnixStream
WebSocket
tonic
aws_sdk
git2
Command::new("curl")
Command::new("wget")
Command::new("powershell")
```

然后把所有裸网络出口替换成：

```rust
safe_network::send(...)
```

这条路线是可行的。

但“一眼一个不漏”的难点在于：

```text
1. 依赖库内部也可能联网。
2. FFI / 动态库可能绕过 Rust 源码搜索。
3. 外部命令可能联网，例如 git / curl / powershell。
4. 构建脚本 build.rs 也可能联网。
5. 插件 / skill / connector 可能下载内容。
```

所以正确做法不是只 grep 一遍，而是三层一起做：

```text
源码扫描
+ 编译期禁止裸网络 API
+ 运行期 safe_network 总闸
```

对你这个项目，网络门禁是比较现实的，因为你已经在做 `NetworkPurpose` 总包装。

## 4. 纯 Rust 工具，本地“读取文件”能不能像联网一样扫描？

**能，而且比网络更适合做门禁。**

需要扫描这些读取入口：

```text
std::fs::read
std::fs::read_to_string
std::fs::File::open
std::fs::OpenOptions::read
std::fs::read_dir
std::fs::metadata
std::fs::canonicalize
tokio::fs::read
tokio::fs::read_to_string
tokio::fs::File::open
walkdir
ignore
glob
globset
notify
```

注意：  
`read_dir` / `walkdir` / `glob` 这种不是读正文，但它们能扫描目录结构，也要算“读取视野”。

推荐门禁：

```text
safe_fs::read_file
safe_fs::read_dir
safe_fs::canonicalize_existing_path
safe_fs::metadata
safe_fs::walk
```

读取规则：

```text
ReadWrite 可读
ReadOnly 可读
Hidden 不可读
硬保护区不可读
Scope 外不可读
```

这个可以做得很彻底。

但要注意：有些“读”不是文件正文，而是：

```text
判断文件是否存在
canonicalize
metadata
列目录
```

这些也要进入门禁体系，否则它能泄露文件是否存在。

## 5. 纯 Rust 工具，本地“创建 / 写入文件”能不能像联网一样扫描？

**也能，而且必须做。**

需要扫描这些写入入口：

```text
std::fs::write
std::fs::File::create
std::fs::OpenOptions::write
std::fs::OpenOptions::append
std::fs::OpenOptions::create
std::fs::OpenOptions::truncate
std::fs::create_dir
std::fs::create_dir_all
std::fs::copy
std::fs::rename
std::fs::remove_file
std::fs::remove_dir
std::fs::remove_dir_all
tokio::fs::write
tokio::fs::File::create
tempfile
```

写入门禁建议分得更细：

```text
safe_fs::write_text_file
safe_fs::append_text_file
safe_fs::create_text_file
safe_fs::create_dir
safe_fs::delete_file
safe_fs::delete_empty_dir
safe_fs::move_file
safe_fs::move_dir
```

写入规则：

```text
普通写入：只能 ReadWrite
创建：父目录必须 ReadWrite
删除：目标必须 ReadWrite，且不能是保护区
移动：来源和目标都必须允许
ReadOnly 不能普通写
Hidden 不能写
.cool / .cool-system 硬保护
```

特殊例外只有一个：

```text
ctool_annotate_markdown
```

它可以对 ReadOnly Markdown 做“只添加批注”的受限写入，但不能删除、替换、重排、改正文。

## 6. 现阶段哪些本地扫描 / 读取是必要的，不能直接关？

按你现在的目标，必要项可以分成三类。

### A. 启动必须保留

这些不能关，否则 CoolCode 自己起不来或不知道安全边界：

```text
读取 LauncherDir\.cool-system\config.toml
读取 LauncherDir\.cool-system\scope.toml
读取 LauncherDir\.cool-system\command.toml

读取 SessionRoot\.cool\config.toml
读取 SessionRoot\.cool\scope.toml
读取 SessionRoot\.cool\command.toml

canonicalize SessionRoot
canonicalize CoolWorkspace
检查 CoolWorkspace 是否存在
```

这些必须存在，但都应该通过 `safe_fs`，并且标记为：

```text
FsPurpose::ReadCoolSystemConfig
FsPurpose::ReadCoolSessionConfig
```

### B. 登录 / 模型调用可能必须保留

这些和 ChatGPT 账号登录、token、会话认证有关：

```text
读取登录状态
读取 token
刷新 token 时写入登录状态
读取基础 config
```

这些不应该给模型看，但程序本身可能需要。  
建议归类为：

```text
FsPurpose::AuthStateReadWrite
```

并限制在固定目录。

### C. 如果保留 resume / compact，就必须保留

这些是 Codex 原生会话历史能力相关：

```text
读取 sessions rollout
写入 sessions rollout
读取 SQLite state
写入 SQLite state
读取 compact 后 history
```

如果你想要：

```text
resume
compact
会话历史
```

那这些不能直接关。

但我建议迁移到：

```text
SessionRoot\.cool\state\sessions
SessionRoot\.cool\state\sqlite
SessionRoot\.cool\state\compact
```

不要继续放在原生：

```text
~\.codex\sessions
```

### D. 可以关掉，或 SafeMode 下默认不应启用

这些不是现阶段必要项：

```text
AGENTS.md 扫描
skills 扫描
plugins 扫描
connectors 扫描
memories 后台整理
update check
telemetry / analytics
feedback
crash report
remote env
watcher
prewarm
外部项目全盘搜索
```

这些都应该 SafeMode 下关掉。

## 最终判断

你这个想法是对的：

```text
网络要有 NetworkPurpose 门禁。
文件也要有 FsPurpose 门禁。
```

但文件比网络更复杂，因为文件能力不只是“读正文 / 写正文”，还包括：

```text
列目录
判断存在
canonicalize
metadata
扫描 glob
watcher
SQLite
日志
缓存
session rollout
```

所以难度确实更大。  
不过它是能系统化解决的，不是完全靠猜。

下一步最推荐做的是：

```text
建立 safe_fs 模块。
先不改所有逻辑。
先把所有 std::fs / tokio::fs / read_dir / canonicalize / OpenOptions 搜出来，列成清单。
然后按必要 / 可关 / 必须迁移 / 必须门禁 四类整理。
```

这一步做完，我们才算真正开始控制“本地文件扫描能力”。
